# ADR-012 — Bun-First Dependency and Build Artifact Policy

**Status:** Accepted
**Date:** 2026-04-26

---

## Decision

This template is **Bun-first**. Bun is the exclusive package manager and script runner. Generated build artifacts and dependency directories are never committed to the repo. `bun.lock` (text format) is always committed.

---

## Context

A reusable template ships to many future projects. Inconsistent toolchain choices create maintenance debt: if one session uses npm and another uses bun, the lockfiles diverge, `node_modules` layouts differ, and CI fails unpredictably. A hard, documented rule eliminates that class of problem.

The three other artifact hygiene rules — never commit `node_modules/`, `.svelte-kit/`, or `build/` — exist because each of those directories is:
- Machine-generated, not human-authored
- Regenerated deterministically from source
- Large (megabytes to gigabytes)
- A frequent source of accidental commits when `.gitignore` is missing

The template shipped without a `.gitignore` for its first two commits. This ADR locks the policy so it cannot regress.

---

## Rules

### Package management

1. **Always use Bun.** Install: `bun install`. Add package: `bun add`. Run script: `bun run`.
2. **Never use** `npm`, `npx`, `pnpm`, or `yarn` — not even for one-off commands.
3. All scripts in `package.json` are authored with Bun in mind. Scripts that call other scripts use `bun run <script>`.

### Lockfile

4. **`bun.lock`** (text format) is committed. It is the source of truth for exact dependency versions and must be reviewed during dependency changes.
5. **`bun.lockb`** (legacy binary format) is gitignored. This repo does not use it. If it appears, do not commit it.

### Protected packages

6. The following packages must not be bumped without explicit approval: `svelte`, `@sveltejs/kit`, `@sveltejs/adapter-*`, `svelte-adapter-bun`, `better-auth`, `esm-env`, `vite`.
7. After any dependency change, diff `bun.lock` and summarize before committing.

### Gitignore — required entries

8. `node_modules/` — never tracked; installed from `bun.lock`.
9. `.svelte-kit/` — generated by SvelteKit on `bun run dev` or `svelte-kit sync`.
10. `build/` — production bundle output; regenerated per deploy.
11. `dist/` — alternative bundle output; same policy.
12. `.env`, `.env.*` (except `.env.example`) — may contain secrets.
13. `bun.lockb` — binary lockfile, not used in this repo.
14. Standard OS/editor noise: `.DS_Store`, `Thumbs.db`, `*.log`, etc.

### Image artifacts — see ADR-009

15. `static/uploads/**/*.webp` (generated sibling files) **are committed** alongside their source images. This is an intentional exception — see [ADR-009](ADR-009-image-pipeline.md).
16. If a future project uses separate output directories (`static/uploads/optimized/`, etc.), those directories must be gitignored.

---

## Consequences

- A `.gitignore` must be present in the repo root. Its absence was the immediate trigger for this ADR.
- `bun.lock` must be staged and committed as part of the initial repo setup.
- `package.json` scripts must use `bun run` when calling other scripts, not `npm run`.
- CI/CD pipelines for projects derived from this template must use `bun install --frozen-lockfile` to verify lockfile integrity.
- Agents working in this repo must run `git ls-files node_modules .svelte-kit build` after any substantive change session and confirm the output is empty.

---

## Alternatives considered

- **pnpm:** Excellent lockfile discipline and workspace support. Rejected because Bun is already chosen as the runtime (`svelte-adapter-bun`) and a second tool would add surface area with no benefit.
- **npm:** Ships with Node. Rejected — produces `package-lock.json` which conflicts with `bun.lock`, and is slower.
- **Committing `node_modules/` for reproducibility:** Rejected — this is a template, not a lockbox. `bun install --frozen-lockfile` provides the same reproducibility guarantee without committing hundreds of megabytes.
- **Ignoring `bun.lock`:** Rejected — without a committed lockfile, `bun install` resolves semver ranges at install time, producing non-deterministic dependency trees across machines and CI runs.
